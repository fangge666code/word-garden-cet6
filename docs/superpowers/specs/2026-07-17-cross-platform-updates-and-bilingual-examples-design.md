# 词间跨平台更新与双语例句改造设计

日期：2026-07-17  
目标版本：1.2.0

## 1. 背景与目标

当前产品已经同时提供 GitHub Pages 网页版、可安装 PWA 和 Android APK，但用户无法清楚知道何时有新版本。现有 3000 词数据中的大部分例句由少量通用模板生成，导致不同单词出现相同或不自然的句子，例如 `The report discusses the role of "longing" in a broader academic context.`。

本次改造包含两个相互独立但共同发布的子系统：

1. 为普通网页、已安装 PWA 和 Android APK 提供适合各自平台的自主更新体验。
2. 重写全部 3000 个单词的英文例句，并为每条例句补充准确的中文翻译。

成功标准是：用户能主动选择是否升级；更新失败不妨碍学习；所有单词都具有自然、可理解、语义匹配的双语例句；既有单词 ID 和学习记录不受影响。

## 2. 已选方案与替代方案

采用“静态版本清单 + 随应用发布的完整双语例句数据”方案。

- 静态版本清单不依赖 GitHub API 的可用性或调用限额，网页和 APK 都能使用同一个公开地址检查版本。
- 双语例句随词库打包，离线时仍然可用，显示速度稳定，也便于自动校验和人工审阅。

未采用的方案：

- 运行时调用 GitHub Releases API：增加网络依赖，并不能改善 PWA 缓存更新体验。
- 运行时使用 AI 生成例句：结果不可复现、离线不可用，并带来费用和内容质量风险。
- 扩充通用模板：仍会产生重复、生硬和语义不匹配的问题。

## 3. 跨平台自主更新

### 3.1 统一版本清单

网站根目录发布一个轻量 JSON 版本清单，至少包含：

- 产品版本名，例如 `1.2.0`；
- 整数版本号，例如 `3`；
- Web 构建标识；
- Android APK 下载地址；
- 简短中文更新说明；
- 发布时间。

客户端请求清单时附加时间戳避免旧缓存，并设置超时。获取失败只显示友好提示，不阻塞任何学习功能。

### 3.2 普通网页版

普通浏览器访问网站时每天最多自动检查一次版本。发现新的 Web 构建后显示非强制弹窗：

- “立即刷新”：清理过期应用缓存、激活新 Service Worker，然后重新载入页面；
- “稍后提醒”：关闭弹窗，24 小时后再提示。

用户也可在首页账号区域手动点击“检查更新”。普通网页不会提示下载 APK。

### 3.3 已安装 PWA

PWA 监听 Service Worker 的新版本安装状态。当新版本已经等待激活时显示“立即升级 / 稍后提醒”：

- “立即升级”：通知等待中的 Service Worker 立即接管，并在控制器变更后只刷新一次；
- “稍后提醒”：继续使用当前版本，24 小时后重新提示。

更新过程不得删除 Supabase 登录状态或本机尚未同步的学习数据。

### 3.4 Android APK

Android 应用启动后每天最多自动检查一次。只有服务器整数版本号大于本机 `versionCode` 时才显示更新弹窗：

- “立即更新”：使用系统外部浏览器打开清单中的 APK 下载地址；
- “稍后提醒”：关闭弹窗，24 小时后再提示。

APK 不进行静默安装，也不强制升级。用户下载后由 Android 系统确认覆盖安装。后续 APK 必须保持相同 application ID 和签名，以便覆盖升级并保留本机数据。

### 3.5 更新状态和边界情况

更新模块只负责平台识别、版本比较、检查频率、延后时间和用户提示，不接触学习业务数据。

- 无网络：继续使用当前版本，并显示“暂时无法检查更新”。
- 清单格式错误：忽略该次结果并记录可诊断错误。
- 下载地址缺失：不显示 Android 更新按钮。
- 用户重复点击：检查和刷新操作在完成前禁用，避免并发。
- 账号未登录：仍可检查应用版本，但现有登录门禁规则保持不变。

## 4. 3000 词双语例句

### 4.1 数据结构与兼容性

每个单词在现有 `example` 字段之外增加 `exampleZh` 字段。保留全部既有 `id`、`word`、`phonetic`、`pos` 和 `meaning`，且不得改变 ID 与单词的对应关系，以免破坏用户已经保存的学习进度。

现有 3000 条英文例句全部重写，包括目前看似可用的少量例句，使整套数据质量和表达风格一致。

### 4.2 内容标准

每个单词必须包含一组匹配的英文例句和中文翻译：

- 英文句子自然完整，优先体现词表中最常用或排在首位的释义；
- 中文句子翻译当前例句的实际含义，不机械罗列词典中的全部释义；
- 英文难度以大学英语六级学习者可理解为准，通常控制在 7 至 18 个英文词；
- 目标词使用合理的词形变化，不能只是放入引号后讨论“这个词”；
- 避免敏感、不适合学习场景或需要额外背景才能理解的内容；
- 禁止使用旧的元语言模板，如“the report discusses the role of”“the article uses”“the word appears frequently”；
- 允许少量结构相似，但不得出现完全相同的英文例句或批量机械替换痕迹。

### 4.3 界面显示

学习卡片翻面后按以下层级显示：单词释义、英文例句、较小且颜色更弱的中文翻译。播放英式发音的现有功能保持不变。

词库总览继续以单词和释义为主，避免 3000 条双语例句让列表过密；本次不新增词库详情弹窗。

### 4.4 数据生产与校验

词库生成脚本不再自行套用四类通用例句模板。完整双语例句作为受版本控制的数据源输入，生成器在缺少任一条时直接失败。

自动校验至少覆盖：

- 总数严格为 3000；
- 每条都有非空 `example` 和 `exampleZh`；
- 英文例句标准化后没有重复；
- 中文翻译包含有效中文字符；
- 不包含旧模板短语、乱码或替换字符；
- 英文长度处于合理范围；
- 目标单词或经审核的词形变化出现在例句中；
- 3000 个既有 ID 与单词的映射保持不变。

自动校验不能代替语义检查。数据完成后按批次人工抽查词义、语法和中英对应关系，并重点复核一词多义、词性混合和不规则词形。

## 5. 模块边界与数据流

更新功能拆分为三个清晰单元：

1. 版本清单读取器：获取、验证并返回服务器版本信息；
2. 平台更新协调器：判断 Web、PWA 或 Android，比较版本并决定可用动作；
3. 更新提示组件：展示更新说明并把用户选择传回协调器。

例句功能拆分为：

1. 受版本控制的双语例句源数据；
2. 生成器，将词表元数据和例句合并为运行时词库；
3. 校验器，阻止缺失、重复或模板化内容进入构建；
4. 学习卡片，只负责呈现 `example` 和 `exampleZh`。

两个子系统互不依赖。更新失败不会影响例句读取，例句数据也不会进入 Supabase 用户进度表。

## 6. 测试与发布

发布前必须完成：

- 版本比较、每天一次检查、稍后提醒和平台分流的单元测试；
- Service Worker 等待、激活和单次刷新的测试；
- Android 仅在更高 `versionCode` 时提供 APK 更新的测试；
- 3000 条双语例句完整性、唯一性、禁用模板和 ID 稳定性测试；
- 首页和学习卡片的界面回归测试；
- 生产构建、PWA 资源校验和 Capacitor Android 同步；
- 桌面浏览器、手机浏览器、已安装 PWA、升级前 Android APK 四类手工验收。

版本发布为 1.2.0。发布顺序为：提交并部署 GitHub Pages、确认版本清单可公网读取、构建并签名 APK、创建 GitHub Release、最后验证旧版 APK 能发现并打开新版下载地址。版本清单只在 APK 下载地址真实可用后指向新 Android 版本，避免用户收到无法完成的升级提示。

## 7. 不在本次范围内

- 应用商店自动更新；
- APK 静默安装或强制更新；
- 在线动态生成例句；
- 修改 Supabase 账号结构或学习进度结构；
- 词库总览的复杂详情页；
- 增加超过当前 3000 条的词量。
