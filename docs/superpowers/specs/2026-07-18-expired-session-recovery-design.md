# 失效登录状态恢复设计

## 问题

Supabase 在刷新令牌失效、被替换或不存在时会返回 HTTP 400。当前客户端把这类响应归入通用云服务错误，导致页面显示“云端服务暂时不可用”，继续提供“立即同步”，并可能因同步失败而阻止退出。实际上云服务仍然在线，需要用户重新登录。

## 目标

- 准确识别 Supabase 的失效刷新令牌响应。
- 将其统一转换为 `AUTH_EXPIRED`，显示“登录状态已失效，请重新登录”。
- 在账号卡片上提供“重新登录”，不再让用户反复点击无效的同步操作。
- 重新登录前把当前账号数据合并保存到匿名本机记录，避免学习进度丢失。
- 登录同一账号后沿用现有“合并本机记录”流程，将本机进度与云端进度合并。
- 不修改 Supabase 项目、数据表、RLS 策略或现有账号数据。

## 错误分类

当 Supabase 响应包含以下认证含义时，无论其状态是 400、401 或 403，都转换为带有 `AUTH_EXPIRED` 代码的中文错误：

- invalid refresh token
- refresh token not found
- refresh token already used
- session not found
- JWT expired 或无效

普通用户名或密码错误仍显示“用户名或密码不正确”；数据库权限、限流和真实网络故障继续使用各自提示。

## 用户流程

1. 自动同步发现会话失效。
2. 账号卡片显示“登录状态已失效，请重新登录”和“重新登录”按钮。
3. 用户点击按钮后，本机学习记录被保留，旧认证信息被移除，首页显示登录表单。
4. 用户使用同一账号登录并选择合并本机记录。
5. 合并后的记录上传到 Supabase，手机和电脑继续同步。

## 测试与发布

- 单元测试覆盖各种刷新令牌失效文案，并确认错误代码为 `AUTH_EXPIRED`。
- 回归测试确认密码错误、数据库权限错误和网络错误没有被误分类。
- 运行全量测试和网页构建。
- 发布到 GitHub Pages，并确认公开页面加载新的客户端文件。
- Android 使用补丁版本发布，避免覆盖现有 1.2.0 安装包。
