# 词间：Supabase 云同步替换设计

## 目标

将尚未上线的 LeanCloud 云同步实现替换为 Supabase，使公开网站继续支持用户名和密码注册、跨手机与电脑同步学习记录，同时保留离线学习、电脑 PWA 安装和后续华为 APK。

## 已确定的产品规则

- 任何人都可以在网站中注册，不要求拥有 GitHub 或 Supabase 账号。
- 学习者只看到“用户名”和“密码”；前端在内部把用户名映射为 Supabase Auth 可接受的唯一登录标识。
- 不提供找回密码。界面在注册时明确提醒用户自行保管密码。
- 同一账号在不同设备登录后，合并本机与云端记录；每个单词以更新时间较新的学习状态为准。
- 新词优先于复习词；不同用户的乱序不同，同一用户的乱序保持稳定。
- 尚未接触的单词不进入学习结果导出。
- 云端暂时不可用时不阻止学习，记录先保存在浏览器本地，恢复连接后再同步。

## 方案选择

采用 Supabase 托管认证和 PostgreSQL 数据库。项目所有者通过现有 GitHub 账号开通 Supabase，但这个 GitHub 授权只用于管理后台，不影响普通学习者注册。

未采用 Firebase，因为项目所有者当前无法稳定完成 Google 登录；未采用自建服务器，因为它需要额外的服务器账号、运维和持续费用。

## 架构与模块

`cloud-config.js` 保存可公开使用的 Supabase Project URL 与 anon key，不保存 service-role key 或任何管理凭据。

新的 `supabase-client.js` 封装注册、登录、恢复会话、读取和写入数据。应用层继续通过稳定的云同步接口工作，避免页面组件直接依赖 Supabase REST 细节。

现有 `account-sync.js` 继续负责本地与云端数据合并；`cloud-sync.js` 调整为使用 Supabase 表结构，但保持对应用层的返回格式不变。

## 数据模型与权限

数据库使用三张表：

- `user_profiles`：每日目标、动效偏好、个人乱序种子和设置更新时间；每位用户一行。
- `word_progress`：单词状态、首次学习时间、最后复习时间、下次复习时间和评分统计；每位用户每个单词一行。
- `daily_records`：每日新词、复习词、评分次数和完成状态；每位用户每天一行。

每张表都含 `user_id`，并启用 Row Level Security。策略只允许 `auth.uid() = user_id` 的用户读取、插入或更新自己的记录。浏览器使用 anon key；管理密钥不会进入代码仓库或网页。

## 用户名登录映射

Supabase Auth 原生使用电子邮箱和密码。网站将规范化后的用户名映射到仅供内部认证的地址，例如 `normalized-name@users.word-garden.invalid`，学习者无需提供真实邮箱。项目关闭邮箱确认后即可直接注册和登录。

用户名仅允许英文字母、数字和下划线，长度 3 至 20 位；比较时不区分大小写，从而避免同名冲突和不可预测的地址转换。该设计与“不提供密码找回”的既定规则一致。

## 同步流程

登录成功后，应用读取三张表中当前用户的数据，与本地记录逐项合并。合并结果立即写回本地，并把有变化的设置、单词记录和每日记录 upsert 到云端。

每次用户评价单词或修改设置后，先完成本地保存，再异步触发同步。同步失败时显示非阻塞提示，不丢弃本地变化。用户退出登录时仅清除登录会话，不删除本机学习记录。

## 错误处理

- 用户名已注册：显示“该用户名已被使用”。
- 用户名或密码错误：使用统一提示，避免泄露账号是否存在。
- 请求频率受限：提示稍后再试。
- 网络或服务异常：说明记录已保存在本机，可继续学习。
- 配置缺失：隐藏或禁用云登录入口，现有离线功能照常可用。

## 测试与上线

自动测试覆盖用户名映射、注册与登录错误翻译、会话恢复、按用户读取、upsert、逐单词合并、离线失败和现有 PWA 行为。配置完成后再做一次真实账号联调：电脑注册并学习，手机登录同一账号，确认学习状态和稳定乱序一致。

最后将替换后的版本推送到 GitHub Pages。公网验证通过后，再让 Android WebView/APK 使用同一线上地址和账号系统。

## 范围边界

本次只替换云端账号和同步服务，不调整学习卡片排版、词表、复习算法、导出规则或网站视觉设计。管理员后台、社交功能、邮箱找回密码和付费系统不在本次范围内。
